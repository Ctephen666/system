# 将香薰配方生成迁移到小智服务端并保留固定配方库兜底

## Goal

取消客户端内对 Qwen 大模型 API 的调用。用户语音请求继续由已连接的小智服务端理解并生成结构化香薰方案；客户端只校验、执行方案，并在服务端未提供有效方案时使用固定香薰配方库，降低额外模型调用产生的时延。现有 16 路继电器驱动及安全关闭语义保持不变。

## Requirements

* 删除香薰模块中的 Qwen/OpenAI 客户端调用、Qwen 配置项和 GUI 中的 Qwen 设置项。
* `aroma.start` 支持接收小智服务端在 MCP 工具调用中提供的可选结构化方案；方案以香型名称、阶段与时长描述，客户端根据现有 `AROMA.CHANNEL_MAP` 转成 16 路执行 pattern。
* 对服务端方案严格校验：香型必须存在于映射中、每阶段时长与总时长受现有配置限制、空/无效阶段不可执行。
* 小智服务端没有提供方案、方案解析失败或校验失败时，使用固定本地香薰库按用户需求选择安全方案；返回结果标明 `xiaozhi_server` 或 `fixed_library` 来源。
* 复用现有 `AromaManager`、`RelayDriver`、串口配置、通道映射和任务取消逻辑，不修改硬件驱动协议或通道关闭行为。
* 固定配方库覆盖至少放松/助眠、专注、提神与默认舒缓场景，并只使用当前配置中存在的香型。
* 更新 MCP 工具描述，使小智服务端明确：优先传递结构化方案，方案缺失时客户端将使用固定库兜底。
* 为新规划器行为增加不访问实际串口或网络的自动化测试。

## Acceptance Criteria

* [ ] 代码和默认配置中不再出现 Qwen、OpenAI 或香薰 API Key 配置。
* [ ] 有效的小智服务端方案被转换为合法的 16 路 pattern 并标记为 `xiaozhi_server`。
* [ ] 缺失、畸形或含未知香型的服务端方案不会触发硬件执行，而是返回固定库方案。
* [ ] 固定库会根据常见需求选择对应配方；无匹配时返回安全默认配方。
* [ ] 现有继电器驱动实现和串口写入语义未改变。
* [ ] 相关测试、编译检查与 Ruff 检查通过。

## Technical Approach

用户语音本来已通过小智连接上传到小智服务端。扩展 `aroma.start` 的 MCP 参数：服务端模型在理解用户需求后可将 `recipe` JSON 与原始 `requirement` 一并传回。客户端的 `AromaPlanner` 只做转换和验证，不再发起第二个模型请求；若无可执行 recipe，则以固定库生成单/多阶段方案。

## Decision (ADR-lite)

**Context**: 客户端二次调用 Qwen 增加网络等待、密钥管理与失败点。

**Decision**: 采用小智服务端 MCP 工具调用承载结构化方案，客户端固定库兜底，不新增直连大模型 API。

**Consequences**: 主路径少一次客户端到模型服务的调用；服务端模型需要遵循工具描述。固定库保证服务端输出不完整或不可用时仍能安全工作，但其个性化程度有限。

## Out of Scope

* 修改小智服务端自身的模型、提示词、鉴权或部署方式。
* 修改继电器协议、串口连接方式或硬件接线。
* 迁移其他非香薰功能使用的模型服务。

## Technical Notes

* 现有实现位于 `src/mcp/tools/aroma/planner.py`，当前会调用 Qwen 后回退本地规则。
* MCP 工具位于 `src/mcp/tools/aroma/_tools.py`，执行生命周期位于 `manager.py`，硬件驱动位于 `driver.py`。
* 用户配置 `config/config.json` 当前包含敏感连接信息；实现不得打印、复制或在文档中暴露这些值。
